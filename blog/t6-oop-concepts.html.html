What are modules?

April 25th, 2015

By now you might have come accross things called "modules" in your quest to become a Ruby master. Maybe you saw them in an example, or in a book on object-oriented programming, or whatever. So what are modules, and why are they imporant?

First and foremost, it's important to remember that Ruby does not allow multiple ineritance. One class can inherit from another class, but it can't inherit from two or three classes at once. Some other languages, such as C++, do allow multiple inheritance, but thankfully Ruby has a neat and powerful way to work around this. I'm going to be using an example given by Matt Stopas in his collection of YouTube videos. Let's say that you have a class loan that has a method called owed_by_month, which calculates how much you need to pay per month on your loan by dividing the balance owed by the maturity date. Now let's say you have another class, bond, that has a method - also named owed_by_month - that returns the amount owed per month by dividing the balance owed by the maturity date. See how both classes contain methods that do the exact same thing? This violates the programming principle of "Don't Repeat Yourself" (DRY).

So how can you NOT repeat yourself? Define a module that takes in the appropriate arguments and contains a method - owed_by_month that calculates how much is owed per month by simply dividing the amount owed by the maturity date. Name the module Amt_owed or something. Calculating the amount owed on something is a fairly common thing to do, right? Lots of different things could use this, not just a loan or a bond. Now go back to your classes and replace the repetitive methods in each of your classes with include Amt_owed. Bam! Now your method owed_by_month can be accessed in both the loan and bond classes. You now have a module that contains a method that you'll need to use in a number of different classes, relieving you from having to write the same method over and over again. Modules allow you to break out of your domain logic, so you don't need to cram everything in your classes. In a way, they're like a global variable that you can use accross classes.

Modules can also be very useful from a namespacing standpoint. If you need to create a class that you know will create a namespace conflict, you can wrap that class inside a module, name the class what you need to name it, and assign the module a name. Then, if you need to create another class in a different file, but your class also presents a namespace conflict, you can simply create another module and give it the same name as the previous one. When you do load both files at runtime, both classes get merged into the same module (because it has the same name in both files). Meaning that you can now call the classes whatever you want without having to worry about a namespace conflict because they're wrapped inside a module!

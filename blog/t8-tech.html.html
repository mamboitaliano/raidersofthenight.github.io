SQL Injection :0

Explanation for newbs

May 9th, 2015

If you're heading into the field of application development, web development, and programming, there's a good chance you might have heard the term "SQL injection" by now. Oooooh hacker stuff! So exciting! But what is SQL injection? SQL injection is a technique used by hackers, both whitehat and blackhat, to attack data-driven applications that use any type of SQL database with the goal of stealing data. It occurs by inserting SQL statements into a data field which exploit a security vulnerability in an application's software. One example of this is when user input is either incorrectly filtered for string literal escape characters embedded in SQL statements, or when user input is not strongly typed and unexpectedly executed.

SQL injection (SQLI) is considered one of the top web application vulnerabilities, and is commonly used in coordinated data-heist attacks. There are 5 main types/subclasses of SQL injection. These include:

  • Classic SQL injection
  • Blind/Inference SQL injection
  • DBMS-Specific SQL injection
  • Compounded SQL injection
    • SQL injection + insufficient authentication
    • SQL injection + DDoS attacks
    • SQL injection + DNS hijacking
    • SQL injection + XSS

The most common method of SQL injection is by exploiting data fields where user input is not properly filtered for escape characters, thus allowing SQL statements to be passed. In essence, this allows a user or hacker to access the database through data input fields, potentially allowing this person to copy or wipe the database, or worse, do both.

Luckily, SQL injection can be avoided. Application developers can use a form of input validation to check the user(or hacker)'s input against a set of pre-defined rules for the type of input. Such rules could include constraints on length, or enforcing strong type user input and syntax. In a nutshell, the more "picky" your user input fields are about the type and length of user input allowable, the safer. The sacrifice, however, is flexibility. Another smart thing to do is to ensure that any user input data is not sent to the SQL interpreter as an SQL query. Failing to safeguard against this could allow a hacker to send SQL commands directly through data fields. To prevent this, application devs can use input validation. Lastly, setting user permission limitations to stringent might also prove to be a smart move.
